# 00-004-04: Update Economic System to Use Inventory

Feature: F004 (Resource Economy System Integration)

## Goal

Rewrite the `EconomicSystem` methods (`consumeResourcesByResidents`, `consumeMaintenanceResources`, `applySpoilage`, `enforceStorageLimits`) to use `economy.Inventory` and `economy.StorageRegistry` instead of directly manipulating the `GameState.Resources` slice. Use proper resource types (grain, vegetables, bread) instead of generic "food".

## Context

The current `EconomicSystem` works directly on `GameState.Resources` with simple string types ("food", "wood", "stone") and hard‑coded spoilage and storage logic. The economy package provides sophisticated inventory management, spoilage calculation based on storage buildings, and storage capacity limits. This workstream replaces the legacy economic logic with calls to economy package functions, while preserving the same external behavior (events, deterministic RNG).

## Dependencies

- WS-004-03: Integrate Inventory into GameState (Inventory field must be present and synced)

## Acceptance Criteria

- [ ] `EconomicSystem` uses `GameState.Inventory` for all resource operations (consumption, spoilage, storage)
- [ ] Resource consumption uses specific economy resource types (`ResourceGrain`, `ResourceVegetables`, `ResourceBread`) not just generic "food"
- [ ] Spoilage calculation uses `economy.ApplySpoilageToInventory` with proper storage building detection
- [ ] Storage limits are enforced via `economy.StorageRegistry` (if a registry is attached)
- [ ] The `EconomicSystem` may optionally hold a `StorageRegistry` instance
- [ ] All events generated by the economic system remain compatible (same types and data fields)
- [ ] Deterministic behavior is maintained (same random seed produces same results)
- [ ] Existing economy tests pass (`go test ./internal/simulation -run TestEconomy`)
- [ ] `go build ./internal/simulation` passes

## Steps

1. **Add StorageRegistry to EconomicSystem**
   - Add `storage *economy.StorageRegistry` field (can be nil)
   - Update `NewEconomicSystem` to accept an optional registry (or create a default one)

2. **Refactor `consumeResourcesByResidents`**
   - Instead of iterating over `state.Resources`, call `Inventory.GetAvailable` for food resource types
   - Determine total food needed (residents count)
   - Consume from inventory using `Inventory.RemoveResource` (prioritize certain food types if needed)
   - Generate consumption event with details (which resource types were consumed)

3. **Refactor `consumeMaintenanceResources`**
   - Map building types to required economy resource types (wood → `ResourceWood`, stone → `ResourceStone`, tools → `ResourceTools`)
   - Calculate total required amounts
   - Remove from inventory via `Inventory.RemoveResource`
   - Generate maintenance event

4. **Refactor `applySpoilage`**
   - Use `economy.ApplySpoilageToInventory` with current game date (convert week/year to `GameDate`)
   - If `StorageRegistry` is available, pass it; otherwise spoilage may use default outdoor storage
   - Collect spoiled amounts per resource type and generate spoilage events
   - Remove the need to manually clean zero‑quantity resources (inventory handles that)

5. **Refactor `enforceStorageLimits`**
   - Use `StorageRegistry` to get capacity per location
   - For each location, check if total quantity exceeds capacity and discard excess
   - Generate storage‑limit events
   - If no registry is present, keep the legacy logic (base + warehouse capacity) as fallback

6. **Update `Update` method**
   - Ensure `GameState.SyncInventory()` is called at start (or assume it's already synced)
   - After inventory modifications, call `GameState.SyncResources()` to keep `Resources` slice updated
   - Preserve the order of operations (consumption → maintenance → spoilage → storage limits)

7. **Update tests**
   - Tests may need to attach a `StorageRegistry` to `EconomicSystem`
   - Update expected resource types in test assertions (e.g., "food" → "grain")
   - Ensure deterministic tests still pass with same seed
