# 00-004-10: Consolidate Storage System ŌĆō Remove Dual Storage and Legacy Resources Slice

Feature: F004 (Resource Economy System Integration)

## Goal

Eliminate the dualŌĆæstorage complexity by making `Inventory` the single source of truth for resources, removing the legacy `Resources` slice from `GameState`. This fixes performance regressions from repeated sync calls, indexŌĆæoutŌĆæofŌĆæbounds bugs in legacy consumption loops, and ensures storage limits apply uniformly.

## Context

The current implementation maintains both an `Inventory` (from the economy package) and a `Resources` slice (legacy simulation resources). The two are kept in sync via `SyncResources` and `SyncInventory` methods, which introduce performance overhead (villageŌĆæbqf) and risk of data inconsistency. The legacy consumption logic in `economy_integration.go` has an indexŌĆæoutŌĆæofŌĆæbounds bug when removing elements while iterating (villageŌĆæhhj). Storage limits are only enforced for food resources because the legacy slice lacks capacity checks (villageŌĆæj1m).

Since phaseŌĆæ4 remediation allows breaking backward compatibility, we can drop the `Resources` slice entirely and migrate all resource operations to use `Inventory`. This simplifies the code, improves performance, and ensures consistent validation and capacity enforcement.

## Dependencies

- WSŌĆæ004ŌĆæ08 (validation and error handling) ŌĆō Inventory must be robust before we rely on it exclusively
- WSŌĆæ004ŌĆæ07 (unknown resource type handling) ŌĆō Legacy save files may contain unknown types that need mapping

## Acceptance Criteria

- [ ] `GameState.Resources` field is removed (or kept as a deprecated computed property for compatibility? ŌĆō we choose removal)
- [ ] `GameState.Inventory` is serialized to JSON (remove `json:"-"` tag) and becomes the primary storage
- [ ] All direct accesses to `gs.Resources` are replaced with inventory methods (`GetAvailable`, `AddResource`, `RemoveResource`)
- [ ] `SyncResources` and `SyncInventory` functions are removed
- [ ] `ConsumeResourceFromState`, `GetAvailableResourceFromState`, `AddProducedResource` are updated to operate solely on `Inventory` (no fallback to legacy slice)
- [ ] The indexŌĆæoutŌĆæofŌĆæbounds bug in `ConsumeResourceFromState` is eliminated (the whole function becomes simpler)
- [ ] Storage limits apply to all resource types because inventory uses the storage registry
- [ ] All existing tests are updated to use inventory; they must pass without relying on `Resources`
- [ ] Save/load tests verify that serialized `Inventory` is preserved across roundŌĆætrips
