# 00-004-08: Fix Validation and Error Handling in Inventory Operations

Feature: F004 (Resource Economy System Integration)

## Goal

Ensure robust validation of resource quantities and types, and propagate errors consistently to prevent data corruption. Eliminate silent error handling in `GameState.AddResource` and other inventory operations.

## Context

Security review identified missing validation of resource quantities (allowing negative values) and inconsistent error handling (village‑qc8, village‑bqx). SRE review flagged silent error handling in `GameState.AddResource` leading to data inconsistency (village‑b0o). Currently:

- `Inventory.AddResource` does not call `Resource.Validate()`
- `Inventory.RemoveResource` does not validate input quantity (negative values cause incorrect behavior)
- `GameState.AddResource` ignores the error from `Inventory.AddResource` (`_ = gs.Inventory.AddResource`)
- No bounds checking for storage capacity for non‑food resources (village‑j1m) – addressed in consolidation workstream

Since backward compatibility can be broken, we can introduce stricter validation and error propagation without worrying about legacy code.

## Dependencies

- Independent (can be done before consolidation)

## Acceptance Criteria

- [ ] `Inventory.AddResource` calls `r.Validate()` and returns an error for invalid resources
- [ ] `Inventory.RemoveResource` rejects negative `quantity` arguments and returns an error
- [ ] `Inventory.TransferResource` validates inputs and propagates errors
- [ ] `GameState.AddResource` returns an error (change signature) and handles it appropriately
- [ ] All callers of `GameState.AddResource` are updated to handle the new error return
- [ ] Tests are added to verify validation rejects negative quantities and unknown types
- [ ] No regression in existing test suite; all inventory and simulation tests pass

## Steps

1. **Add validation to `Inventory.AddResource`**
   - Open `internal/economy/inventory.go`
   - At the start of `AddResource`, call `r.Validate()`; if false, return an error describing the issue
   - Also validate that `r.Quantity >= 0` (already covered by `Validate` but double‑check)
   - If a storage registry is attached, keep the existing capacity checks

2. **Add validation to `Inventory.RemoveResource`**
   - At the beginning of `RemoveResource`, check `quantity >= 0`; if not, return an error
   - Also ensure `quantity` is not `NaN` or `Inf` (though float64, we can check with `math.IsNaN` and `math.IsInf`)

3. **Update `Inventory.TransferResource`**
   - Validate `quantity >= 0` and propagate errors from `RemoveResource`/`AddResource`

4. **Change `GameState.AddResource` signature**
   - Open `internal/simulation/state.go`
   - Change `func (gs *GameState) AddResource(r Resource)` to `func (gs *GameState) AddResource(r Resource) error`
   - Remove the `_ = gs.Inventory.AddResource` line; instead call `err := gs.Inventory.AddResource(...)` and return the error
   - If `gs.Inventory` is nil, simply append to `gs.Resources` (legacy mode) and return `nil`

5. **Update all callers of `GameState.AddResource`**
   - Search for `.AddResource(` in the simulation package and update each call site to handle the error (log, ignore, or propagate)
   - Most callers are in tests; adjust tests accordingly (they usually expect success, so can call `require.NoError` or `assert.Nil`)

6. **Add validation tests**
   - In `inventory_test.go`, add test cases for negative quantities, unknown resource types, and capacity violations
   - In `state_test.go`, add tests that verify error propagation from `AddResource`

7. **Run verification commands**
   - Build economy and simulation packages to ensure no compilation errors
   - Run inventory tests to confirm validation works
   - Run simulation tests to ensure error handling does not break existing behavior
