# 00-004-12: Add Infinity and NaN Validation to Resource.Validate and Inventory Methods

Feature: F004 (Resource Economy System Integration)

## Goal

Ensure robust validation of resource quantities by rejecting infinite (`Inf`) and Not-a-Number (`NaN`) values in `Resource.Validate()` and inventory operations. This prevents infinite loops, unexpected behavior, and data corruption in economic calculations.

## Context

Security review identified a P1 blocking finding (village‑6ub): `Resource.Validate()` only checks `Quantity >= 0` but does not reject positive/negative infinity (`math.IsInf`). This allows infinite quantities to be added to the inventory, which can cause infinite values in calculations (e.g., `ComputeValue`) and potentially break game logic. `Inventory.RemoveResource` already validates NaN/Inf, but `Validate()` does not, creating an inconsistency.

Since backward compatibility can be broken, we can add stricter validation without affecting legitimate game data.

## Dependencies

- Independent (can be executed anytime)

## Acceptance Criteria

- [ ] `Resource.Validate()` returns `false` for infinite (`math.IsInf`) and NaN (`math.IsNaN`) quantities
- [ ] `Inventory.AddResource` (which calls `Validate`) rejects invalid quantities with appropriate error messages
- [ ] `Inventory.RemoveResource` continues to reject NaN/Inf quantities (already implemented)
- [ ] `Inventory.TransferResource` propagates validation errors
- [ ] Tests are added to verify validation rejects NaN, +Inf, and -Inf quantities
- [ ] Existing tests pass; no regression in inventory or simulation functionality

## Steps

1. **Update `Resource.Validate()`**
   - Open `internal/economy/resources.go`
   - Add `math` import if not already present
   - Modify `Validate` method to check `math.IsNaN(r.Quantity) || math.IsInf(r.Quantity, 0)`
   - Return `false` if either condition is true

2. **Add validation to `Inventory.AddResource` (defense in depth)**
   - Already calls `Validate`; no change needed unless we want to add explicit check (optional)

3. **Update `Inventory.TransferResource` error propagation**
   - Ensure errors from `RemoveResource` and `AddResource` are properly propagated (should already be the case)

4. **Add test cases**
   - In `resources_test.go`, add `TestResource_Validate_NaNInf` that tests NaN, +Inf, -Inf quantities
   - In `inventory_test.go`, add `TestInventory_AddResource_NaNInf` that verifies `AddResource` rejects invalid quantities
   - Update existing tests if they rely on the old behavior

5. **Run verification commands**
   - Build economy package to ensure no compilation errors
   - Run validation-specific tests
   - Run full inventory test suite

## Code

Snippet for updated `Validate`:

```go
import "math"

func (r Resource) Validate() bool {
    if !IsValidType(r.Type) {
        return false
    }
    if r.Quantity < 0 || math.IsNaN(r.Quantity) || math.IsInf(r.Quantity, 0) {
        return false
    }
    return true
}
```

Snippet for test:

```go
func TestResource_Validate_NaNInf(t *testing.T) {
    r := Resource{Type: ResourceGrain, Quantity: math.NaN()}
    if r.Validate() {
        t.Error("Validate should reject NaN")
    }
    r = Resource{Type: ResourceGrain, Quantity: math.Inf(1)}
    if r.Validate() {
        t.Error("Validate should reject +Inf")
    }
    r = Resource{Type: ResourceGrain, Quantity: math.Inf(-1)}
    if r.Validate() {
        t.Error("Validate should reject -Inf")
    }
}
```
