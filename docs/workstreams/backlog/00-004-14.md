# 00-004-14: Eliminate Silent Error Handling in Economic Loop

Feature: F004 (Resource Economy System Integration)

## Goal

Replace silent error handling in economic simulation loops with explicit logging and proper error propagation. This improves debuggability, prevents undetected data inconsistencies, and addresses SRE's P1 blocking finding (village‑ayj).

## Context

SRE review flagged silent error handling in `economy.go` where errors from `Inventory.RemoveResource` are ignored without logging. For example:
- `CalculateFoodConsumption` ignores `RemoveResource` errors (line 87)
- `calculateBuildingMaintenance` ignores `RemoveResource` errors (line 167)
- `enforceStorageLimits` ignores `RemoveResource` errors (line 322)

These silent failures can mask underlying issues (invalid quantities, insufficient resources, NaN/Inf values) and make debugging difficult. Since backward compatibility can be broken, we can add logging and adjust error handling without affecting external interfaces.

## Dependencies

- Independent (can be executed anytime)

## Acceptance Criteria

- [ ] All `RemoveResource` error returns in `economy.go` are logged at appropriate severity (WARN for unexpected errors, DEBUG for expected cases like "resource not present")
- [ ] `CalculateFoodConsumption` logs a DEBUG message when `RemoveResource` returns "insufficient quantity" (expected) and a WARN message for other errors
- [ ] `calculateBuildingMaintenance` logs a DEBUG message for "insufficient quantity" and WARN for other errors
- [ ] `enforceStorageLimits` logs a WARN message for any `RemoveResource` error (should not happen under normal operation)
- [ ] No change to function signatures or external behavior (errors are still ignored for simulation continuity, but logged)
- [ ] Log messages are structured (key‑value pairs) and include relevant context: resource type, quantity, location, operation
- [ ] Existing tests pass; no regression in simulation behavior

## Steps

1. **Audit silent error locations**
   - Search for `err != nil` patterns in `economy.go`, `social.go`, `production.go`, `events.go`
   - Identify each location where the error is ignored or silently consumed

2. **Add logging helpers**
   - Optionally add a package‑level helper function `logRemoveResourceError(err error, rt economy.ResourceType, quantity float64, location string)` that logs at appropriate level based on error content
   - Or inline log statements with consistent format

3. **Update `CalculateFoodConsumption`**
   - At line 87, replace `// Ignore error (e.g., resource not present)` with a conditional log:
     - If error message contains "insufficient quantity", log at DEBUG/INFO level
     - Otherwise log at WARN level

4. **Update `calculateBuildingMaintenance`**
   - At line 167, add similar conditional logging

5. **Update `enforceStorageLimits`**
   - At line 322, replace `// Should not happen` with a WARN log (any error here indicates a validation bug)

6. **Check other simulation files**
   - Review `social.go`, `production.go`, `events.go` for similar patterns (e.g., `AddResource` errors)
   - Add logging where appropriate

7. **Run verification commands**
   - Build simulation package to ensure no compilation errors
   - Run economic and social tests to verify logging does not break existing behavior
   - Optionally run integration tests to ensure no performance regression

## Code

Snippet for `CalculateFoodConsumption`:

```go
removed, err := state.Inventory.RemoveResource("global", rt, float64(foodNeeded))
if err != nil {
    // Log expected "insufficient quantity" as DEBUG, others as WARN
    if strings.Contains(err.Error(), "insufficient quantity") {
        log.Printf("DEBUG: operation=CalculateFoodConsumption resource=%s needed=%d message=\"resource not present, skipping\"", rt, foodNeeded)

}
```
