# 00-006-03: Code Audit for Modern Go Best Practices

Feature: F006 (Go Modernization)

## Goal

Analyze the codebase for compliance with modern Go best practices (Go 1.18+) and produce a prioritized report with recommendations for improvements. Focus on error handling, concurrency patterns, generics opportunities, API design, and standard library usage.

## Scope Files

- `internal/` - Core application code
- `cmd/` - Command-line interfaces
- `go.mod` - Dependencies and Go version

## Dependencies

- WS-006-02 (Add Modern Go Linting Tools) - Linters will help identify some issues

## Acceptance Criteria

- [ ] **Error Handling Analysis**: Review error patterns, usage of `errors.Is`, `errors.As`, error wrapping with `fmt.Errorf("%w")`, sentinel errors
- [ ] **Generics Opportunities**: Identify places where generics could improve type safety, reduce duplication, or simplify code
- [ ] **Concurrency Patterns**: Audit goroutine usage for leaks, proper synchronization with `sync` package, channel patterns
- [ ] **API Design Review**: Evaluate interface design, coupling, package boundaries, and public API stability
- [ ] **Standard Library Usage**: Check for use of deprecated functions, newer packages (e.g., `slices`, `maps`, `cmp`), and modern alternatives
- [ ] **Performance Patterns**: Identify potential performance issues (allocations, inefficient loops, etc.)
- [ ] **Prioritized Report**: Create report with high/medium/low priority recommendations, estimated effort, and impact
- [ ] **Actionable Items**: Each recommendation should include specific code examples and suggested changes

## Context

The codebase has 800+ Go files and may have been developed over time with evolving Go practices. Modern Go (1.18+) introduces generics, improved error handling, and new standard library packages that can improve code quality, safety, and performance. This audit will identify the most valuable improvements without requiring a full rewrite.

## Input Files

1. Entire Go codebase (`internal/`, `cmd/`)
2. `go.mod` for dependency analysis
3. Existing test files to understand usage patterns

## Steps

1. **Error Handling Analysis**
   - Use `grep`/`rg` to find error patterns
   - Check for proper error wrapping vs string concatenation
   - Identify sentinel error definitions and usage
   - Review error handling in deferred functions

2. **Generics Opportunities**
   - Identify repetitive code with similar logic for different types
   - Look for `interface{}` or `any` usage that could be typed
   - Check container types (slices, maps) with manual type assertions
   - Identify common algorithms that could be genericized

3. **Concurrency Audit**
   - Search for `go` keyword usage
   - Review `sync` package usage (WaitGroup, Mutex, RWMutex, Pool)
   - Check for channel patterns (buffered vs unbuffered, closing)
   - Identify potential race conditions with `go vet -race`

4. **API Design Review**
   - Analyze package boundaries and dependencies
   - Review interface definitions and implementations
   - Check for exported vs unexported identifiers
   - Evaluate struct design and method receivers

5. **Standard Library Usage**
   - Check for deprecated functions (`ioutil`, `strings.Title`, etc.)
   - Identify opportunities for `slices`, `maps`, `cmp` packages
   - Review `context` usage patterns
   - Check `time` package usage for monotonic time

6. **Performance Analysis**
   - Identify high-allocation patterns (string concatenation in loops, etc.)
   - Review slice/map pre-allocation opportunities
   - Check for inefficient loops or nested operations

7. **Report Generation**
   - Categorize findings by priority (high/medium/low)
   - Estimate effort for each recommendation (SMALL/MEDIUM/LARGE)
   - Provide specific code examples before/after
   - Suggest sequencing of changes

8. **Optional: Tool-assisted analysis**
   - Use `staticcheck` with specific rules
   - Use `gocritic` for code review
   - Use `errcheck` for unhandled errors

## Code

### Example analysis command for error patterns
```bash
# Find error returns without wrapping
grep -r "fmt.Errorf(" --include="*.go" | grep -v "%w" | head -20

# Find sentinel error definitions
grep -r "errors.New(" --include="*.go" | head -20

# Find goroutine launches
grep -r "^[[:space:]]*go " --include="*.go" | wc -l
```

### Example recommendation format
```markdown
## High Priority: Error Wrapping
**File**: `internal/simulation/engine.go:123`
**Issue**: Error concatenation without wrapping loses context
**Before**: `return fmt.Errorf("failed to process: " + err.Error())`
**After**: `return fmt.Errorf("failed to process: %w", err)`
**Effort**: SMALL (mechanical change)
```
