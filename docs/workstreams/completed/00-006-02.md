# 00-006-02: Add Modern Go Linting Tools

Feature: F006 (Go Modernization)

## Goal

Integrate modern Go linting tools (golangci-lint) into the project to improve code quality, catch common issues, and enforce consistent coding standards. Establish automated linting in CI/CD pipeline.

## Scope Files

- `.golangci.yml` - Configuration for golangci-lint with appropriate rules
- `Makefile` - Add `lint` target that runs golangci-lint
- `.github/workflows/lint.yml` - GitHub Actions workflow for automated linting
- `README.md` - Update with linting instructions
- Optional: `scripts/install-golangci-lint.sh` - Installation helper script

## Dependencies

- WS-006-01 (Update Go Version) - Should be completed first for version compatibility

## Acceptance Criteria

- [ ] `.golangci.yml` configuration file created with appropriate rules for the project
- [ ] `Makefile` contains `lint` target that runs `golangci-lint run ./...`
- [ ] `make lint` passes on existing codebase (or exceptions configured for legacy code)
- [ ] GitHub Actions workflow `.github/workflows/lint.yml` runs on PRs and main branch
- [ ] Linting workflow fails if linting errors are detected
- [ ] README.md updated with linting instructions for developers
- [ ] All new code submissions will be automatically linted
- [ ] Common issues like unused variables, format inconsistencies, and potential bugs are caught

## Context

The project currently uses basic Go tools (`go vet`, `go fmt`) but lacks comprehensive static analysis. Modern Go projects benefit from golangci-lint which aggregates multiple linters (staticcheck, revive, gosec, etc.) with configurable rules. This improves code quality, reduces bugs, and maintains consistency across the codebase.

## Input Files

1. Existing `Makefile` - To add lint target
2. Existing `.github/workflows/go.yml` - Reference for workflow structure
3. Project codebase - To configure appropriate linter rules

## Steps

1. **Create golangci-lint configuration**
   - Create `.golangci.yml` file in project root
   - Configure linters appropriate for project size and complexity
   - Set reasonable timeouts and exclusions
   - Enable useful linters: staticcheck, govet, revive, gosec, errcheck, etc.
   - Disable linters that are too noisy for existing codebase initially

2. **Add lint target to Makefile**
   - Add `lint` target to `Makefile` that runs `golangci-lint run ./...`
   - Consider adding `lint-fix` target for auto-fixable issues
   - Update `.PHONY` list to include `lint`

3. **Create GitHub Actions lint workflow**
   - Create `.github/workflows/lint.yml`
   - Use `actions/setup-go@v5` with Go 1.25
   - Install golangci-lint using official action `golangci/golangci-lint-action@v6`
   - Configure to run on PRs and pushes to main
   - Cache golangci-lint results for performance

4. **Test linting locally**
   - Install golangci-lint (via curl script or package manager)
   - Run `make lint` to see initial issues
   - Adjust `.golangci.yml` configuration to handle project-specific needs
   - Fix trivial issues or add exclusions for legacy code

5. **Update documentation**
   - Add section to `README.md` about linting
   - Include instructions for installing golangci-lint
   - Explain how to run linting and fix common issues

6. **Optional: Create installation script**
   - Create `scripts/install-golangci-lint.sh` for easy setup
   - Script downloads appropriate version for platform

## Code

### Example .golangci.yml
```yaml
run:
  timeout: 5m
  modules-download-mode: readonly

linters:
  disable-all: true
  enable:
    - errcheck
    - gosimple
    - govet
    - ineffassign
    - staticcheck
    - typecheck
    - unused
    - gosec

issues:
  exclude-use-default: false
  max-issues-per-linter: 0
  max-same-issues: 0

linters-settings:
  gosec:
    excludes:
      - G204 # Subprocess launched with variable

output:
  format: colored-line-number
```

### Makefile addition
```makefile
.PHONY: build test run clean dev coverage vet fmt lint

lint:
	golangci-lint run ./...

lint-fix:
	golangci-lint run --fix ./...
```

### GitHub Actions workflow (.github/workflows/lint.yml)
```yaml
name: Lint

on:
  push:
    branches: [ main, master ]
  pull_request:
    branches: [ main, master ]

jobs:
  golangci:
    name: lint
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-go@v5
        with:
          go-version: '1.25'
      - name: golangci-lint
        uses: golangci/golangci-lint-action@v6
        with:
          version: latest
          args: --timeout=5m
```
